# UnMeet 客户画像 & 会议深度研究

> 2026年7月

## 一、五大客户画像

### 画像 1：远程优先科技创业公司

| 维度 | 描述 |
|------|------|
| **规模** | 20-100 人 |
| **团队结构** | 工程 + 产品 + 设计 + 少量商务，扁平组织 |
| **技术栈** | Slack + Zoom/Google Meet + Notion/Linear + GitHub |
| **会议工具** | 混用 Zoom、Google Meet、Slack Huddle——没有统一平台 |
| **关键人物** | CTO/VP Engineering（感觉团队被会议淹没但没数据证明） |
| **付费意愿** | 🟢 高——VC 投的钱，愿意为效率买单 |
| **年预算** | $500-5000/年（工具类） |
| **决策人** | CTO / VP Engineering |

**为什么他们是理想客户：**
- 跨平台是刚需（他们没有单一会议平台）
- 工程师文化 = 痛恨会议 = 愿意量化问题
- 扁平组织 = 每个人都在开会 = 数据量大
- 有预算，决策快（一个人说了算）

### 画像 2：数字创意 / 广告代理商

| 维度 | 描述 |
|------|------|
| **规模** | 15-80 人 |
| **团队结构** | 客户管理 + 创意 + 制作 + 策略 |
| **技术栈** | Teams/Zoom + Slack + Figma + 邮件 |
| **会议工具** | Zoom 为主，部分用 Teams |
| **关键人物** | 创始人/Managing Director（盯着利润率） |
| **付费意愿** | 🟢 高——利润率薄，每一分钱都要算 |
| **年预算** | $300-3000/年 |
| **决策人** | MD / Operations Director |

**为什么他们是理想客户：**
- 内部会议不创造收入（只有客户会议是 billable 的）
- 利润率通常只有 15-25%，会议成本直接吃利润
- MD 对"这个会让我少赚了多少钱"极其敏感
- 团队小，一两个人就能推动采购决策

### 画像 3：专业服务公司（律所/咨询/会计）

| 维度 | 描述 |
|------|------|
| **规模** | 10-200 人 |
| **团队结构** | 合伙人 + 高级顾问 + 初级顾问/律师/会计师 |
| **技术栈** | Teams 为主 + Outlook + 内部系统 |
| **会议工具** | Microsoft Teams 为主 |
| **关键人物** | Managing Partner（盯着 billable hours） |
| **付费意愿** | 🟢 极高——时间是唯一的商品，会议直接等于丢钱 |
| **年预算** | $1000-10000/年 |
| **决策人** | Managing Partner / COO |

**为什么他们是理想客户：**
- 按小时计费 = 每个非 billable 的会议都有明确的金钱代价
- 合伙人级别的时薪 $200-500，一小时内部会 = 丢掉 $500
- 已经习惯了追踪 billable hours，会议追踪是自然延伸
- 行业痛点明确：知道内部会议太多但无法量化

### 画像 4：中大型企业的工程/产品部门

| 维度 | 描述 |
|------|------|
| **规模** | 部门 30-300 人（公司可能 500-5000 人） |
| **团队结构** | 工程总监 → 工程经理 → IC；产品 VP → PM → Designer |
| **技术栈** | Teams 或 Zoom + Jira + Confluence + Slack |
| **会议工具** | Teams（公司强制）或 Zoom |
| **关键人物** | VP Engineering / Head of Product |
| **付费意愿** | 🟡 中等——需要向上申请预算，但金额小容易批 |
| **年预算** | $2000-15000/年 |
| **决策人** | VP + 采购部门 |

**为什么他们是理想客户：**
- 工程团队会议抱怨是经久不衰的话题
- VP 需要数据来说服 CEO/CPO "少点会"
- 预算流程长但一旦采购会长期续费
- 需要对标数据（"同行工程团队平均开会 10h/周，你们是 18h"）

### 画像 5：中国科技/互联网公司

| 维度 | 描述 |
|------|------|
| **规模** | 30-200 人 |
| **团队结构** | 技术 + 产品 + 运营 + 市场，管理层级较多 |
| **技术栈** | 飞书或企业微信/腾讯会议 + 内部 OA |
| **会议工具** | 飞书/腾讯会议为主（单平台） |
| **关键人物** | CEO/CTO/HRD（关注人效和加班文化） |
| **付费意愿** | 🟡 中等——需要看到明确 ROI 才买单 |
| **年预算** | ¥3000-30000/年 |
| **决策人** | CTO/HRD（人效是 HR 的 KPI） |

**挑战：**
- 单平台用户：飞书本身有会议功能，但不会说出成本
- 中国公司会议文化更根深蒂固（"不开会=不重视"）
- 需要本地化定价（¥58/月而非 $8）
- 切入点不是"少开会"而是"提升人效"（HR 话术）

## 二、各画像的会议特征深度分析

### 画像 1：远程科技创业公司

**会议结构：**

| 会议类型 | 频率 | 时长 | 参与人 | 月人均小时 |
|---------|------|------|--------|----------|
| Daily Standup | 5x/周 | 15-30min | 5-10人 | 6-12h |
| Sprint Planning | 2周1次 | 2-4h | 工程全队 | 4-8h |
| 1:1 | 1x/周 | 30min | 每个 manager | 2h |
| Product Review | 1x/周 | 60min | 跨职能 5-12人 | 4h |
| All-Hands | 1x/月 | 60min | 全员 | 1h |
| Retro | 2周1次 | 60min | 每小队 | 2h |
| 临时 sync | 不定 | 30-60min | 2-5人 | 4-8h |
| **合计** | | | | **23-37h/月** |

**典型痛点：**

1. **Standup 蠕变**：本来 15 分钟，不知不觉变成 30 分钟。15 人 standup = 每天浪费 3.75 小时
2. **跨时区会议地狱**：旧金山 × 伦敦 × 新加坡 = 总有人要早上 6 点或晚上 10 点开会
3. **产品评审超时**：没有预读材料，前 20 分钟在"了解背景"
4. **临时 sync 爆炸**：Slack 上一句"quick sync?"变成 45 分钟会
5. **工程师被碎片化**：每天 3-5 个短会，最长连续专注时间不到 2 小时

**怎么改：**

| 问题 | 改法 | 怎么算省钱 |
|------|------|----------|
| Standup 蠕变 | 拆成 5 人 pod standup（非全员） | (15-5)人 × 10min × 5天 = 8.3h/周 |
| 产品评审超时 | 强制预读材料 + 30min 硬上限 | 8人 × 30min = 4h/周 |
| 临时 sync 爆炸 | 设"深度工作时段"（上午不开会） | 5h/周恢复为连续工作时间 |
| 跨时区会议 | 改为异步更新 + 每周 1 次 overlap | 5人 × 1h × 2次 = 10h/周 |
| **合计月度节省** | | **~107 小时/月** |

**省钱金额**（按 $75/h 均薪）：

> 107h × $75 = **$8,025/月** ≈ **$96,300/年**

### 画像 2：广告/创意代理商

**会议结构：**

| 会议类型 | 频率 | 时长 | 参与人 | 月人均小时 |
|---------|------|------|--------|----------|
| 客户周报 | 1x/周/客户 | 30-60min | 3-5人 | 12-20h |
| 内部创意评审 | 2-3x/周 | 60-90min | 3-8人 | 16-24h |
| 制作 handoff | 按项目 | 30min | 2-3人 | 2-4h |
| 新业务 pitch | 1-2x/月 | 60min | 4-8人 | 2-4h |
| 团队站会 | 3-5x/周 | 15min | 全员 | 3-5h |
| 项目复盘 | 按项目 | 60min | 项目组 | 2-4h |
| **合计** | | | | **37-61h/月** |

**典型痛点：**

1. **内部会议不赚钱**：客户会议 = billable，内部会议 = 纯成本
2. **创意评审反复**：第一轮没对齐方向，后面全浪费
3. **"再改一版"文化**：每次评审都加一轮修改 → 更多评审 → 更多会
4. **Account Manager 过载**：AM 一天 5-6 个客户会 + 3 个内部会 = 没时间做 actual work
5. **老板不知道哪个项目在亏钱**：客户 billing 够 cover 团队时间吗？不知道

**怎么改：**

| 问题 | 改法 | 怎么算省钱 |
|------|------|----------|
| 创意评审反复 | 给评审设"决策人"（一个人拍板，不投票） | 减少 1 轮评审 = 6人 × 1.5h = 9h/项目 |
| AM 过载 | 合并客户例会 + 减少内部 sync | 4h/周 |
| 无产出的站会 | 改成 Slack 异步 check-in | 5人 × 15min × 5天 = 6.3h/周 |
| 项目复盘 | 只复盘甲方投诉的或有重大问题单 | 2h/周 |
| **合计月度节省** | | **~85 小时/月** |

**省钱金额**（按 $60/h 均薪，利润率 20%）：

> 85h × $60 = $5,100/月成本 → 如果用于 billable 工作可创造 $6,375/月收入（加价 25%）
> **年影响：$76,500 收入潜力**

### 画像 3：专业服务（律所/咨询）

**会议结构：**

| 会议类型 | 频率 | 时长 | 参与人 | 月人均小时 |
|---------|------|------|--------|----------|
| 客户会议（billable）| 5-15x/周 | 30-60min | 2-5人 | 30-50h |
| 内部案件讨论 | 2-3x/周 | 30-60min | 3-6人 | 8-16h |
| 合伙人会议 | 1x/周 | 60-90min | 合伙人 | 4-6h |
| 业务拓展 | 2-4x/月 | 60min | 2-3人 | 2-4h |
| 培训/知识分享 | 1-2x/月 | 60min | 全员可选 | 1-2h |
| **非 billable 会议合计** | | | | **15-28h/月** |

**典型痛点：**

1. **合伙人时薪极高**：$300-500/h 的合伙人在内部会上 = 每分钟烧 $5-8
2. **初级员工被拉着参加不必要的会**：客户会带 3 个 junior "学习"——其实 2 个就够了
3. **内部案件讨论没产出**：讨论了一小时，没做任何决定
4. **业务拓展会不可追踪**：不知道 BD 会花了多少时间、带来了多少客户
5. **培训出席率低**：安排了但没人来，或者来了也没学进去

**怎么改：**

| 问题 | 改法 | 怎么算省钱 |
|------|------|----------|
| 合伙人在非 essential 会议 | 设规则：合伙人只参加决策会，信息同步会看纪要即可 | 2h/周 × $400 = $800/周 |
| 初级员工过度参与 | 客户会限制在核心 2 人，其他人异步看笔记 | 1人 × 8h/周 × $150 = $1,200/周 |
| 内部讨论无产出 | 强制"每个内部讨论必须有结论+行动人"（不满足不开） | 减少 30% 时长 = 5h/周 × $200均 = $1,000/周 |
| **合计月度节省** | | **$12,000/月** |

**省钱金额：**

> **$12,000/月 ≈ $144,000/年**
> 按 15 人律所算：人均节省 $9,600/年。相当于一个初级律师半年的工资。

### 画像 4：企业工程/产品部门

**会议结构：**

| 会议类型 | 频率 | 时长 | 参与人 | 月人均小时 |
|---------|------|------|--------|----------|
| Standup | 3-5x/周 | 15-30min | 工程小队 | 4-8h |
| Sprint仪式（plan+review+retro） | 2周循环 | 4-6h | 工程队 | 8-12h |
| 跨部门对齐 | 2-3x/周 | 30-60min | PM+Eng+Design | 6-12h |
| 1:1 | 1x/周 | 30min | 每个manager | 2h |
| All-Hands / Town Hall | 1x/月 | 60min | 部门全员 | 1h |
| 架构评审 | 1-2x/周 | 60min | 3-6人 | 4-8h |
| 临时 debug/同步 | 不定 | 15-60min | 2-4人 | 3-6h |
| **合计（工程师）** | | | | **18-30h/月** |
| **合计（PM）** | | | | **30-55h/月** |
| **合计（EM）** | | | | **35-60h/月** |

**典型痛点：**

1. **PM 被会议活埋**：55h/月开会 = 每天 2.5h+，剩下的时间不够写 PRD
2. **EM 的日历像俄罗斯方块**：全是背靠背的 30min 1:1 + sync
3. **工程师的碎片化**：上午 9-10 standup → 10-11 架构会 → 2-3 跨部门对齐 → 真正写码的时间只有下午 3-6 的三个小时
4. **越多 PM = 越多会**：每个 PM 都要拉工程师 sync → 工程师的会议量与 PM 数量成正比
5. **跨部门会议是传话游戏**：A 和 B 开完 → B 和 C 开 → C 和 D 开。一个信息传 4 层

**怎么改：**

| 问题 | 改法 | 怎么算省钱 |
|------|------|----------|
| PM 被埋 | PM 设定"不开会时段"（每天 4h 连续深度工作） | PM 产出提升 30%（定性） |
| EM 背靠背 | 1:1 集中到周二/周四 | 周一三五无会议 = 可以做 technical work |
| 工程师碎片 | 设 No Meeting Day（如周三） | 1天×团队人数 = 大量连续编码时间 |
| 跨部门传话 | 用文档替代同步会：决策记录写进 Notion | 3个会 × 5人 × 30min = 7.5h/周 |
| **合计月度节省** | | **~120 小时/月** |

**省钱金额**（按 $80/h 均薪）：

> 120h × $80 = **$9,600/月** ≈ **$115,200/年**

### 画像 5：中国科技公司

**会议结构：**

| 会议类型 | 频率 | 时长 | 参与人 | 月人均小时 |
|---------|------|------|--------|----------|
| 晨会/站会 | 5x/周 | 15-30min | 团队 | 5-10h |
| 周会 | 1x/周 | 60-120min | 部门 | 4-8h |
| 月度经营会 | 1x/月 | 2-4h | 管理层 | 2-4h |
| 项目评审 | 2-3x/周 | 60-90min | 跨部门 4-10人 | 8-18h |
| 1:1 | 1x/周 | 30min | 管理者 | 2h |
| 全员会 | 1x/月 | 60-90min | 全员 | 1.5h |
| 汇报/述职 | 按需 | 30-60min | 2-5人 | 3-6h |
| **合计** | | | | **25-50h/月** |

**典型痛点：**

1. **周会是最浪费时间的形式**：每人读上周做了什么——完全可以用文档替代
2. **月度经营会准备成本巨大**：中层花 1-2 天做 PPT，只为 2 小时汇报
3. **"领导在场"现象**：领导参加的会，每个人都要发言表态——本来 30 分钟的会拖到 90 分钟
4. **跨部门推诿会**：出了 bug，产品怪开发，开发怪测试，测试怪需求。——开了 3 次会也没结论
5. **加班开会**：晚上 8 点以后或周末的会，美其名曰"奋斗"

**怎么改：**

| 问题 | 改法 | 怎么算省钱 |
|------|------|----------|
| 周会读报告 | 改为飞书文档异步评论 + 只讨论需要决策的议题 | 10人 × 1.5h = 15h/周 |
| 月度经营会 | 只汇报例外（异常数据），正常的不汇报 | 5人 × 3h + 准备时间 15h = 30h/月 |
| 领导在场拖时长 | 设规则：需要领导决策的议题前 15 分钟讨论完，领导只参加决策环节 | 5人 × 40min × 3次 = 10h/周 |
| 跨部门推诿 | 用 RACI 矩阵替代 blame game 会议 | 6人 × 1h × 2次 = 12h/月 |
| **合计月度节省** | | **~172 小时/月** |

**省钱金额**（按 ¥120/h 均薪）：

> 172h × ¥120 = **¥20,640/月** ≈ **¥247,680/年**

## 四、MVP 应该先打哪个

**画像 1：远程科技创业公司。** 理由：

1. 他们混用 Zoom + Meet + Slack——跨平台是刚需
2. PH / HN / Reddit 能直接触达
3. 决策快（一个人拍板）
4. 内部有工程师文化，数据驱动，愿意为效率工具买单
5. 省钱数据对他们有冲击力（"我们的 VC 钱被会议烧了 $96K/年"）

**第二步扩散到 画像 3：专业服务**——用律所/咨询的省钱数据（$144K/年）做案例营销。
